# Docker Volumes

## Interview Question

What are Docker volumes and how do they differ from bind mounts and tmpfs?

## Interview Answer

Docker volumes are the preferred mechanism for persisting data generated by containers. They are managed entirely by Docker, stored in a part of the host filesystem managed by Docker (`/var/lib/docker/volumes/`), and can be shared across multiple containers. Bind mounts map a specific host path directly into the container, while tmpfs mounts create temporary in-memory storage that disappears when the container stops. For Rust backends, volumes are essential for database storage, log files, and sharing build artifacts between containers.

## Follow-up Questions & Answers

### Q1. How do you create and use a named volume?

**Interview Answer**

Create a volume with `docker volume create my-data` and mount it with `-v my-data:/app/data`. The volume persists independently of container lifecycles — removing the container doesn't delete the data. List volumes with `docker volume ls` and inspect with `docker volume inspect`. For a Rust backend storing database files, named volumes ensure data survives container restarts and rebuilds without manual host path management.

---

### Q2. When would you use a bind mount instead of a named volume?

**Interview Answer**

Bind mounts are ideal during development when you need to share source code or configuration files between host and container. For example, mounting your Rust `src/` directory into a build container enables live code reloading without rebuilding the image. Bind mounts are also used when the container needs access to specific host paths like SSL certificates. The tradeoff is that bind mounts depend on the host directory structure and don't work across different operating systems.

---

### Q3. What is a tmpfs mount and when is it appropriate?

**Interview Answer**

A tmpfs mount stores data in the host's memory only — it's never written to disk and disappears when the container stops. Use it for sensitive data like API keys that shouldn't persist, or for high-performance temporary files like Rust build caches. Mount with `--tmpfs /app/tmp:rw,size=100m`. It's faster than disk-based volumes but uses RAM, so size limits are important to prevent memory exhaustion.

---

### Q4. How do you share data between multiple containers?

**Interview Answer**

Multiple containers can mount the same named volume simultaneously for read/write access. Use `--volumes-from` to inherit all volumes from another container, or specify the volume name explicitly in each container's run command. For a Rust backend and a sidecar logging container, both can mount the same log directory volume. Be aware that concurrent writes require proper file locking — Rust's `std::fs` doesn't provide cross-process locking, so use advisory locks or external coordination.

---

### Q5. How do you back up and restore Docker volumes?

**Interview Answer**

Back up a volume by running a temporary container that mounts it and creates a tar archive: `docker run --rm -v my-data:/data -v $(pwd):/backup alpine tar czf /backup/backup.tar.gz /data`. Restore by reversing the process — mount the backup directory and extract into a volume. For production Rust databases, coordinate backups with the database's own backup tools rather than just copying files, as the database may have files in an inconsistent state.

---

### Q6. What happens to volume data when a container is removed?

**Interview Answer**

Named volumes persist after container removal — `docker rm` doesn't delete them. You must explicitly run `docker volume rm` to remove volume data. Anonymous volumes (created without a name) are also removed by default unless you pass `-v` flag during creation. Bind mounts are never removed by Docker since they reference host paths. For production Rust deployments, always use named volumes for critical data and be cautious with the `-v` flag during cleanup.

---

### Q7. How do you inspect volume permissions and ownership?

**Interview Answer**

Use `docker volume inspect <volume-name>` to see the mountpoint and driver details. Inside the container, check with `ls -la` and `stat` on the mounted path. Volume permissions can be problematic when a non-root process in the container tries to write to a volume owned by root. For Rust containers running as non-root, either pre-create the volume with correct ownership, use `chmod` in the Dockerfile, or use bind mounts with specific UID/GID mappings.

---

### Q8. What is the difference between `-v` and `--mount` syntax?

**Interview Answer**

The `-v` (or `--volume`) syntax is compact: `-v host:container:options` but creates the host path automatically if it doesn't exist for bind mounts. The `--mount` syntax is more verbose but explicit — it uses key-value pairs and won't create missing directories. `--mount` also supports additional options like `volume-type`, `readonly`, and `consistency`. Docker recommends `--mount` for production use because its explicit behavior prevents accidental directory creation on the host.

---

### Q9. How do volumes perform compared to the container's writable layer?

**Interview Answer**

Named volumes and bind mounts have essentially the same I/O performance as direct host filesystem access since they bypass the container's copy-on-write layer. The writable layer uses overlay2 or similar drivers, which add overhead for writes due to copy-on-write semantics. For Rust applications with heavy I/O like database servers or file processors, using a volume is significantly faster than writing to the container's filesystem. This is one reason PostgreSQL and other databases always recommend mounted volumes.

---

### Q10. How do you clean up unused volumes and prevent disk bloat?

**Interview Answer**

Run `docker system prune -volumes` to remove all unused volumes, or `docker volume prune` specifically. Use `docker volume ls -f dangling=true` to see which volumes are unattached. In CI/CD pipelines, always clean up volumes after builds. For Rust development, the `target/` directory mounted as a bind mount can grow to several gigabytes — periodically run `cargo clean` inside the container or recreate the volume. In production, use monitoring to track volume growth and set up automated cleanup policies.
