# How Do You Implement Idempotency?

## Interview Question

How do you implement idempotency in a distributed system?

## Interview Answer

The standard approach is to use an idempotency key — a unique identifier generated by the client or server and stored alongside the operation result. On subsequent requests with the same key, the system returns the previously stored result instead of re-executing the operation. This can be implemented using Redis with atomic SET NX operations or a database table with unique constraints. The key must have a TTL to prevent unbounded storage growth, and the entire check-and-execute sequence must be atomic to prevent race conditions. For database operations, you can also use unique constraints or conditional inserts to enforce idempotency at the storage level.

## Follow-up Questions & Answers

### Q1. Walk me through the exact Redis implementation of an idempotency check in Rust.

**Interview Answer**

In a Rust Axum handler, you receive the idempotency key from the request header. You use Redis `SET key value NX EX 3600` (set-if-not-exists with 1-hour expiry). If the key already exists and has a value of "processing", you return HTTP 409 Conflict. If the key exists with a completed result, you return the cached response. If SET NX succeeds, you process the request, store the result with `SET key result EX 3600`, and return it. You use the `redis` crate with `cmd("SET").arg(&key).arg(&result).arg("NX").arg("EX").arg(3600)` for atomic operations. This ensures only one request processes a given key, and retries get the cached result.

### Q2. What are the failure modes of the Redis-based idempotency approach?

**Interview Answer**

If Redis crashes after SET NX succeeds but before the result is stored, the key exists with "processing" status but no result — subsequent retries will get a 409. The mitigation is to use a transaction or Lua script to atomically check and store. If Redis loses the key (eviction, restart without persistence), retries will re-execute the operation. Using Redis AOF with `everysec` fsync reduces this risk. If the application crashes after processing but before returning to the client, the client retries and gets a 409 — you need a cleanup worker to resolve stale "processing" states. Network partitions between the service and Redis can cause the idempotency check to fail, in which case you might fall back to database-level checks.

### Q3. How do you implement idempotency at the database level instead of Redis?

**Interview Answer**

Create an `idempotency_keys` table in PostgreSQL with columns for the key, status, result, and created_at timestamp. Before processing, use a transaction: `INSERT INTO idempotency_keys (key, status) VALUES ($1, 'processing') ON CONFLICT DO NOTHING`. If the insert returns affected rows = 1, you own the key and proceed. If 0, a duplicate exists — check the status and return the cached result. After processing, `UPDATE idempotency_keys SET status = 'completed', result = $2 WHERE key = $1`. The unique constraint on the key column prevents duplicates at the database level, making this approach resilient even if Redis is unavailable. Periodically clean up old keys with a cron job.

### Q4. How do you handle the case where the idempotency key is valid but the operation partially fails?

**Interview Answer**

Partial failures are the trickiest scenario. If a payment succeeds but a subsequent notification fails, you cannot simply retry without charging twice. The solution is to store intermediate states: "pending", "processing", "completed", "failed". If processing fails, store the error state so retries return the error instead of re-executing. For multi-step operations, use the saga pattern — each step is idempotent and has a compensating action. If step 3 of 5 fails, you roll back steps 1 and 2, and the idempotency key marks the operation as "rolled_back" so retries do not re-execute any step.

### Q5. What is the difference between request-level and operation-level idempotency keys?

**Interview Answer**

Request-level keys ensure that the exact same HTTP request is processed once — the key is tied to a specific request (e.g., `POST /orders` with key `abc-123`). Operation-level keys ensure that the same logical operation is processed once, even if the request details change slightly (e.g., the same order intent with different timestamps). Request-level is simpler to implement and sufficient for most use cases. Operation-level requires semantic understanding of what makes two requests "the same" and is useful when clients may retry with slightly different payloads. For a payment API, the operation-level key might be based on (user_id, amount, recipient, timestamp_bucket) rather than a random UUID.

### Q6. How do you implement idempotency for batch operations?

**Interview Answer**

Batch operations need a batch-level idempotency key plus individual item keys. The batch key ensures the entire batch is processed once, while individual keys allow partial retries if some items succeed and others fail. In a Rust service processing a batch of Kafka messages, you might use a transaction that processes all items atomically, with a unique constraint on each item's idempotency key. If the batch partially fails, you can mark individual items as failed and let the consumer retry only those items. This is more complex than single-request idempotency because you need to track the state of each item within the batch independently.

### Q7. How does idempotency work with eventual consistency in databases?

**Interview Answer**

When using eventual consistency (e.g., async replication to read replicas), an idempotency key might not be visible to all replicas immediately. If you check the key on a replica that has not yet received the write, you might process a duplicate. Solutions include: reading the idempotency key from the primary (adds latency but guarantees consistency), using a quorum read (read from a majority of replicas), or accepting the risk and adding a compensating mechanism. In practice, for critical operations like payments, you always check the idempotency key against the primary database or a strongly consistent store like Redis with replication.

---

### Q8. What is the relationship between idempotency keys and database unique constraints?

**Interview Answer**

Database unique constraints enforce idempotency at the storage level — an INSERT with a duplicate key will fail, preventing duplicate records. Idempotency keys at the application level provide a more flexible mechanism: they can store the full response, handle different failure modes, and provide user-friendly error messages. The best practice is to use both: a unique constraint on the idempotency key table as a safety net, and application-level idempotency logic for proper error handling. For example, an INSERT with ON CONFLICT DO NOTHING combined with an application-level check before the INSERT provides defense in depth.

### Q9. How do you test idempotency implementations?

**Interview Answer**

Test idempotency by sending the same request multiple times and asserting that the operation is executed exactly once. Use concurrent requests (e.g., 100 parallel calls with the same idempotency key) to verify race condition handling. Verify that the stored response is identical for all retries. Test edge cases: key expiration during processing, Redis/database unavailability during the idempotency check, and partial failures. In Rust, use `tokio::spawn` with multiple concurrent tasks sending identical requests to an Axum test server, then verify side effects (database rows, payments) occurred exactly once using assertions against the test database.

### Q10. Can you implement idempotency without any external state store?

**Interview Answer**

Yes, using deterministic operation IDs and conditional logic. If the operation is a pure function of the idempotency key (e.g., "set value X for key Y"), you can use the database's upsert (INSERT ON CONFLICT DO UPDATE) to naturally deduplicate. For state-changing operations, you can derive the operation from the key itself — if the key encodes the intent (e.g., `payment_user123_20240115_50USD`), the server can reconstruct what was requested and skip already-completed operations. However, this approach is fragile and limits flexibility. External state stores (Redis, PostgreSQL) are almost always preferred because they provide reliable, general-purpose deduplication regardless of operation semantics.
