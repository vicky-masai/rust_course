# What is a self-referential struct, and why is it difficult?

## Interview Question

What is a self-referential struct, and why is it difficult?

## Interview Answer

"A self-referential struct contains pointers to its own fields. Moving the struct invalidates those pointers, so Rust requires `Pin` or specialized crates to make this safe."

---

## Follow-up Questions & Answers

### Q1. Why can't you safely move a self-referential struct?

**Interview Answer**

When a struct contains a pointer to one of its own fields, moving the struct changes the base address but not the pointer. The pointer now points to the old, invalid memory location. In C, this is undefined behavior. Rust prevents this at compile time by making self-referential structs impossible to create safely without `Pin`.

---

### Q2. What is `Pin<P>` and how does it solve the problem?

**Interview Answer**

`Pin<P>` guarantees that the pointed-to value will not be moved in memory. When you pin a value, you get a `Pin<&mut T>` that prevents the value from being moved, keeping self-references valid. `Pin` is essential for async/await because futures generated by async functions are self-referential across await points.

---

### Q3. What is the `Unpin` trait?

**Interview Answer**

`Unpin` is an auto-trait implemented for types that can be safely moved even when pinned. Most types implement `Unpin` by default. Only types that opt out (like async futures) are affected by pinning. `Pin::into_inner` requires `T: Unpin`, allowing you to extract the value when you know it's safe to move.

---

### Q4. How do async functions create self-referential structs?

**Interview Answer**

When you use `.await` in an async function, the compiler generates a state machine future where variables across await points are stored in the same struct. If you borrow a local variable before an `.await` and use it after, the future becomes self-referential. This is why `async fn` always returns an `Unpin`-requiring future.

---

### Q5. What is the `pin-project` crate?

**Interview Answer**

`pin-project` provides a procedural macro that safely projects `Pin<&mut Self>` to inner fields without violating pin invariants. It generates the correct `unsafe` code to ensure that projecting a pin to a field is safe. This is essential for implementing `Future` manually or creating structs with pinned fields.

---

### Q6. Can you create a self-referential struct without `Pin`?

**Interview Answer**

Yes, using heap allocation and indices instead of pointers. For example, `Vec<T>` with indices instead of references, or `SlotMap` with stable keys. Alternatively, `ouroboros` or `self_cell` crates provide macros that safely create self-referential structs by controlling allocation and pinning internally.

---

### Q7. How does `Pin` interact with the drop glue?

**Interview Answer**

When a pinned value is dropped, `Pin` ensures the value is not moved before `drop` is called. The `Drop` trait receives `&mut self` as a pinned reference. For `!Unpin` types, `Pin::drop` is used instead of direct `drop`, maintaining the pin invariant throughout the destructor's execution.

---

### Q8. What are common use cases for self-referential structs?

**Interview Answer**

Async futures are the most common case—every `async fn` generates a self-referential state machine. Other cases include zero-copy parsers that borrow from input buffers, doubly-linked lists where nodes reference each other, and event processing systems where events borrow from a buffer while being processed.

---

### Q9. How does `Pin` affect API design?

**Interview Answer**

APIs that return pinned values (like `async fn`) must document pinning requirements. Consumers need to understand that pinned values can't be moved. This affects how you store futures—you might need `Box::pin` for heap allocation or `tokio::pin!` for stack pinning. The `Future` trait itself takes `Pin<&mut Self>` in `poll`.

---

### Q10. What are the limitations of `Pin`?

**Interview Answer**

`Pin` doesn't prevent all forms of unsafe interaction—you can still create self-referential structs unsafely with `unsafe` code. It only guarantees that pinned values won't be moved by safe code. The main limitation is complexity: developers must understand pinning to write correct async code, and `!Unpin` types require careful handling throughout the API.
